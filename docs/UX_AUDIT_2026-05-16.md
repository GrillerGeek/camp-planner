# UX Audit — 2026-05-16

End-to-end code walkthrough of the production app. Focus: real workflows
(create templates → create trip → apply templates → use the trip).

Approach: read every route + component for the affected flows, identify
gaps with file:line specificity. Did NOT use Playwright/Chrome because
Google blocks OAuth in automation-driven browsers; a code walkthrough
proved more thorough since it surfaces edge cases (empty states, viewer
paths, error branches) that wouldn't be hit in a single manual session.

## Severity legend

- **P0** — broken or actively misleading; user-visible functional bug
- **P1** — significant friction or feature gap on a common path
- **P2** — polish, inconsistency, or rough edge

---

## P0 — Broken or misleading

### A1. Packing: no template picker (the user's flagged gap)
**Where:** `app/dashboard/trips/[tripId]/packing/components/PackingListClient.tsx:243`
plus `lib/queries/packing.ts:370` (`autoPopulateFromTemplates`).

The packing UI only exposes a single "Auto-populate from templates" button.
It calls `autoPopulateFromTemplates` which iterates **all** templates whose
seasons overlap the trip and merges them. There is no UI to pick a
specific template. The user cannot say "apply only my Weekend Tent Camping
template to this trip."

Tasks has the right pattern — a modal listing each template
(`TaskListClient.tsx:546-635`) with per-template Apply buttons. Packing
should adopt the same pattern.

### A2. Packing: template-apply button disappears once any item exists
**Where:** `PackingListClient.tsx:241` — gated on `isPlanner && total === 0`.

After auto-populating (or adding even one manual item), the button is
hidden entirely. You can never apply an additional template to an existing
list. If you forgot to apply your "Winter add-ons" template before adding
manual items, you have to delete everything and start over.

### A3. Template `is_essential` is silently dropped on apply
**Where:** Schema mismatch.

- `packing_template_items.is_essential` exists (`002_packing_lists.sql`)
- `trip_packing_items` has **no** `is_essential` column
- `autoPopulateFromTemplates` (`lib/queries/packing.ts:407`) doesn't even
  read `item.is_essential` when building the insert

So the essential flag on templates is purely decorative — it never makes
it to a real packing list. The trip-side UI also has zero references to
`is_essential` anywhere (`grep` confirms).

### A4. Trip-type filtering on templates is non-functional
**Where:** `lib/queries/packing.ts:findMatchingTemplates` + `lib/types/trip.ts`.

`PackingTemplate.trip_types` is one of `tent | rv | cabin | backpacking`,
and the create-template form lets you set it. But `Trip` has no
`trip_type` field — only `name, destination, start_date, end_date,
campsite_info, notes`. So no trip carries a type, so no template can
ever be filtered by trip_type. The UI affordance is misleading.

This is already documented in `SPEC-004b.2`, but it has been mis-classified
as "medium" priority — it's actually a P0 because users will spend time
tagging templates with types that do nothing.

## P1 — Significant friction

### B1. Packing template items: no edit, only delete
**Where:** `app/dashboard/templates/components/TemplateItemsEditor.tsx`.

To fix a typo in a template item ("flashilght" → "flashlight"), you must
delete and re-add. Task template items have inline edit
(`TaskTemplateItemsEditor.tsx:50-71`); packing template items do not.
Direct inconsistency.

### B2. Two-step template creation (form → edit page to add items)
**Where:** `TemplateForm.tsx:54` redirects to `/edit` after creating the
shell. Same pattern in `TaskTemplateForm`.

User must click "Create Template" then land on an entirely different page
to start adding items. A combined creation flow (or at minimum a clearer
"you'll add items next" hint on the create button) would reduce confusion.

### B3. Packing template-item form has no `notes` field
**Where:** `TemplateItemsEditor.tsx:24-30` — the local `newItem` state has
no `notes` field, even though the underlying `addTemplateItem` accepts
notes and the schema column exists.

So `notes` on packing template items can never be set via the UI. Probably
removed during development and not restored.

### B4. Dashboard trip card: past + non-completed trips clutter the main view
**Where:** `app/dashboard/page.tsx` + `lib/queries/trips.ts:getTripsForUser`.

`getTripsForUser` returns **all** trips regardless of date or status. A
trip that ended last weekend but wasn't marked "completed" stays mixed
with upcoming trips on the dashboard. Only completed trips are filtered
out (via the separate `/history` page).

Suggested: split the dashboard into Upcoming / Past sections, OR auto-
suggest completion for trips past their end_date.

### B5. Viewers cannot check off ANY packing items (even assigned to them)
**Where:** `PackingListClient.tsx:399` — `disabled={!isPlanner}`, tooltip
"Only planners can mark items packed".

The SPEC-004 retro decided viewers are read-only on packing. This is
actually documented behavior but it conflicts with the product premise
that someone bringing assigned gear should be able to mark it packed.
A guest invited as viewer who's responsible for "tent" can't check it
off when they pack it.

Product question worth resolving: is this intended? If so, document it
prominently. If not, allow check-off on items where
`assigned_to === current_user.id`.

### B6. Recipe library: tap-to-edit, no view-only mode
**Where:** `app/dashboard/recipes/page.tsx:60` — recipe cards link directly
to `/edit`.

Tapping a recipe takes you into edit mode. No way to "view" a recipe
without entering edit. For non-owners this is also confusing — the page
loads but writes will be RLS-rejected.

Also matches `SPEC-005b.4` deferred item (no search/filter/pagination).

### B7. Recipe-from-template-item duplication risk
**Where:** `RecipeForm.tsx` flow.

Two recipes can share the same name with no warning. Already documented
in `SPEC-005b.5` but worth surfacing.

### B8. Trip status auto-progression
**Where:** Schema has `status: planning | active | completed` on trips,
but the lifecycle is manual. A trip with `start_date = yesterday` is
still "planning" forever unless the user explicitly transitions it.

Suggested: an "active" auto-transition when current_date is within
start_date..end_date, with a banner on the trip page. Or surface this
as a one-click "Mark trip active/completed" on the trip header.

## P2 — Polish, inconsistency, rough edges

### C1. Template list page: no item count preview
The template card on `/dashboard/templates` shows item count, but the
items themselves aren't summarized. A "shelter, cooking, clothing, +3"
preview line would help users pick the right template before clicking.

### C2. Packing template items: no reorder
`sort_order` is set on insert but no drag-reorder UI exists. The display
order is creation order, which is awkward for sorted-by-category lists.

### C3. Task template description field is single-line input
`TaskTemplateItemsEditor.tsx:101` uses `<input type="text">` for
description, capping at one line visually. The DB stores text. A
`<textarea>` would let users add multi-line notes.

### C4. "Auto-populate from templates" button — confusing wording
The button says "Auto-populate" but really means "Apply matching season
templates." Users likely expect "auto" to mean smart AI-driven (since the
meal planner uses AI). Rename to "Apply template" once a picker exists.

### C5. Trip detail page: no Members link from the readiness cards section
Members link only lives in TripHeader (newly added). A "Members" card
alongside the four readiness cards could surface presence count + quick
access.

### C6. Empty packing list: "Add items manually or auto-populate from
your templates" — but no link to templates page if user has none.
`PackingListClient.tsx:366`.

### C7. Recipe card on Recipe Library: tags use camp-forest color but
recipe-page meal cards use the same color. Inconsistent — meal-context
tags should differentiate.

### C8. Inventory: no expired-item filter on the inventory page itself
Grocery generation correctly excludes expired items, but the
`/dashboard/inventory` page shows them mixed in (per the file structure,
not deeply audited). Worth verifying.

### C9. AI Meal Suggestions button: visible on trips with 0 meals OR
500 meals, no notion of "enough suggestions for now." After applying
3 suggestions, the panel auto-removes them, which is good — but no
"all done" state.

### C10. Trip member email lookup case-insensitivity (already fixed
2026-05-16) — but the error message says "no user with that email" without
suggesting they check spelling.

## Recommended fix order

If we want one consolidated b-spec for this audit, group as:

**`SPEC-002c`: Template apply parity + correctness (P0 batch)**
- Add packing template picker modal (mirror tasks)
- Allow re-applying templates after items exist
- Add `is_essential` column to `trip_packing_items` + propagate from template
- Either remove `trip_types` from template form OR add trip_type to Trip
- Add edit affordance to packing template items
- Restore `notes` field on packing template-item form

**`SPEC-002d`: Trip lifecycle + dashboard hygiene (P1 batch)**
- Split dashboard into Upcoming / Past
- Auto-suggest completion on trips past end_date
- Decide + document viewer check-off behavior (B5)

**Defer to existing b-specs**
- B6/B7 already covered by SPEC-005b
- Multi-assignee in SPEC-004b.1
- trip_type matching in SPEC-004b.2
